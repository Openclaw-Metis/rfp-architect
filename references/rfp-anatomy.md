# RFP 結構與必備章節（write 與 review 共用知識庫）

本檔定義一份委外開發 / 資訊系統採購 RFP（建議書徵求說明書）的**標準骨架**與每一章該寫什麼。write 模式用它決定要產出哪些章節；review 模式用它判斷既有 RFP 缺了什麼。

## 0. 採購文件家族（先用對文件）

| 文件 | 用途 | 採購階段 | 重點 |
|---|---|---|---|
| RFI（資訊徵求） | 蒐集市場 / 廠商資訊 | 最早期、無購買承諾 | 概括性資訊 |
| **RFP（建議書徵求說明書）** | 徵求綜合解決方案 | 需求已明確 | 技術方案＋實施＋報價，評綜合價值 |
| RFQ（報價徵求） | 規格標準品比價 | 需求標準化 | 價格、數量、交期 |
| SOW（工作說明書） | 得標後執行藍圖 | **決標後**、具法律拘束力 | 具體任務 / 交付 / 驗收標準 |

- 需求未明確就硬發 RFP → 收到無法比較的提案。需要創新解法卻用 RFQ 比價 → 把採購窄化成價格競爭。
- RFP 的範疇是「寬廣、描述性」；SOW 的範疇是「細部、規定性」。**驗收標準是 SOW 的核心，但其可驗收性必須在 RFP 的需求中就埋好**。

## 1. 十一項必備章節（write 必填；review 逐項檢查）

1. **執行摘要與專案願景** — 公司與專案簡述、願景與目標、成功定義。
2. **業務背景與目標** — 為何採購、現狀指標（使用量 / 處理時間 / 滿意度）、痛點、3 / 6 / 12 個月量化目標、階段範圍。
3. **工作範疇（Scope of Work）** — 任務、責任、活動；**明確劃分「包含」與「不包含」**以防範疇蔓延；保留創新空間（夠細以引導，不過度限制）。
4. **功能需求** — 系統「應該做什麼」；以**表格＋優先級（必要 / 應有 / 加分）**列出；每條可驗證、對應業務目標。
5. **技術與非功能需求** — 整合架構（要求架構圖）、部署模式、效能 / 安全 / 可用性 / 易用性 / 可擴充 / 可維護等品質屬性，逐項量化。
6. **資通安全與法規遵循** — 資安分級、禁用清單、資料落地、在地化（見 `clause-library.md`）。
7. **服務水準協議（SLA）與效能指標** — 可量化承諾（可用率、回應時間、解決率）＋未達標罰則。
8. **商業模式與定價** — 付費方式（買斷 / 訂閱 / 用量）、標準化報價範本以利比較。
9. **評選標準與配分** — 加權計分矩陣、各面向權重、決標方式（最有利標 / 評分及格最低標）（見 `taiwan-procurement.md`）。
10. **預算與時程里程碑** — 預算範圍、期望時程、里程碑切分、（政府採購）法定等標期。
11. **契約條款與投標須知** — 履約 / 驗收 / 智財 / 保固條款前置揭露、提交格式 / 頁數限制 / 釋疑窗口 / 聯絡方式。

> 三支柱：**工作範疇、功能需求、評選標準**。其餘章節都圍繞這三者服務。

## 2. 需求工程（第 4、5 章的底層方法）

- **功能需求**＝系統「做什麼」，使用者直接感受。以 use case 捕捉（actor＋goal），具體、可驗證、對應業務目標。
- **非功能需求**＝系統「做得多好」，橫切品質屬性，最常被忽略卻決定上線成敗。類別：效能、安全性、可用性、易用性、可擴充性、可維護性，**逐項給量化門檻**（例：同時 1,000 人在線、可用率 99.9%、API < 200ms）。
- **三層需求**：業務需求（為何做）→ 功能需求（做什麼）→ 非功能需求（多好）。
- 黃金準則：**每條需求都可測試、可對應驗收標準**。寫不出驗收標準的需求＝還沒想清楚的需求。SRS 結構＝簡介 / 整體描述 / 系統特性與需求 / 其他需求。

## 3. 撰寫流程與最佳實務（write 模式的前置動作）

1. **發包前先做利害關係人訪談＋需求鎖定**：對齊產品負責人 / PM / 工程師 / 終端使用者，把功能記錄到 user story、為每個功能定義可衡量驗收標準，發包前凍結基準線。
2. **清晰、簡明語言**：減少誤解、收到可比較提案；善用範本但必客製化。
3. **在 RFP 內就揭露評選標準**：透明配分讓廠商據以調整提案；（政府採購）規則公告後不得變更。
4. **管理時程**：訂提案截止日、釋疑期限、里程碑；給足準備時間。

## 4. 現代趨勢（加分章節 / review 時的進階建議）

- **成果導向採購**：以目標說明書（SOO，2–3 頁、定義廣泛目標與期望成果）取代規定式 SOW；適合敏捷 / 迭代開發。
- **彈性化 RFP**：分階段、聚焦業務成果 / 文化契合 / 概念驗證（PoC）；契約模型如「按衝刺固定計價」「成果型協議」。
- **AI 輔助**：AI 協助草擬與比對；採購 AI 系統時須要求結構化 PoC、精確測試情境與一致評估量表。
- 但成果導向需要成熟的甲方治理；需求穩定、合規極嚴的情境，傳統固定範疇 RFP 仍較穩健。
