# 委外失敗模式 → RFP 控制點對照

**來源聲明**：本頁的案例屬公開、業界廣為討論的政府 / 大型資訊專案（一般產業知識），**非本 skill 所附研究報告的內容**；引用個案細節時請另行查證。本頁的價值在於「失敗模式 → 可寫進 RFP 的控制點」這層對照——review 模式可用它強化 finding 的「為什麼重要」，write 模式可用它檢查控制點是否已落點。

## 對照表

| 失敗模式 | 典型徵兆 | 對應 RFP / 契約控制點 |
|---|---|---|
| 需求未凍結就大規模簽約 | 範疇邊界模糊、邊做邊改、變更失控 | 需求基準線、正式變更請求（CR）流程、分階段 gate、範疇含 / 不含（見 `clause-library.md` I、`rfp-anatomy.md` §2） |
| 整合 / 測試太晚 | 各模組各自完成、上線才發現兜不起來 | 端到端整合測試、壓力 / 負載測試、上線演練與平行測試列為里程碑（見 `rfp-anatomy.md` §2 系統轉換） |
| 治理責任分散、無單一問責 | 多頭馬車、決策卡關、無人負責 | 單一問責窗口、甲方產品負責人、可於時限內裁決之發起人、週期性 steering meeting（見 `clause-library.md` K） |
| 長約鎖死、難以更換廠商 | 資料 / 原始碼被綁、議價弱勢 | 分階段採購、可替換模組、退場 / 資料返還 / 刪除銷毀 / 移交（見 `clause-library.md` J） |
| 廠商低價搶標後靠變更回收 | 得標價偏低、變更請求不斷加價 | 範疇含 / 不含、變更估價規則、付款與 gate 綁里程碑、揭露實際團隊與 PoC（見 `clause-library.md` K、付款矩陣） |
| 以精緻提案得標、實作團隊不符 | 投標團隊 ≠ 履約團隊 | 要求揭露實際參與人員與資歷、關鍵人員替換須同等資格並經同意 |

## 知名公開案例（僅作失敗模式之例示，細節請自行查證）

- **美國 Healthcare.gov（2013 上線危機）**——整合多個後端、負載與整合測試不足、缺單一系統整合問責，常被引為「整合 / 測試太晚 + 治理分散」的代表。
- **英國 NHS NPfIT（國家級醫療 IT 計畫）**——規模過大、需求未凍結、由上而下推動，最終解散重組，是「需求未凍結就大規模簽約 + 長約鎖死」的代表。
- **澳洲 Queensland Health 薪資系統**——需求反覆、測試不足、成本大幅超支，是「需求 / 範疇失控 + 測試太晚」的代表。
- **英國 GDS / GOV.UK（正面對照）**——以使用者為中心、敏捷迭代、分階段交付，顯示成果導向 / 可替換模組可降低上述風險（呼應 `rfp-anatomy.md` §4 成果導向採購）。
