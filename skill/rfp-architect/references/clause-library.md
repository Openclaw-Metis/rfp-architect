# 台灣在地條款庫（可直接貼入 RFP）

這些是國際 RFP 範本通常沒有、但台灣委外案不可或缺的條款。write 模式視標的勾選貼入；review 模式檢查既有 RFP 是否遺漏。**請依個案調整等級、門檻與罰則數值。**

## A. 資通安全分級

> 本案系統依「資通系統防護需求分級原則」評估為【普／中／高】等級；得標廠商應落實對應等級之存取控制、加密、日誌稽核與弱點管理措施，並於各階段提供佐證。

- 分級越高，控制越嚴。涉及個資或關鍵業務者通常為中／高級。

## B. 廠商資安資格

> 得標廠商應簽署資訊安全承諾書與保密同意書，並提供 ISO 27001 驗證證書或完成資安自評查核表。

## C. 禁用產品與人員（台灣特有）

> 本案不得使用大陸廠牌之資通訊產品；廠商執行團隊成員不得為大陸地區人民。

- 公部門與關鍵基礎設施標準配備；遺漏可能於審標爭議或履約後衍生資安事件。

## D. 安全開發生命週期（SSDLC）

> 廠商應遵循安全軟體開發生命週期，於需求、設計、開發、測試、上線各階段執行安全檢核並留存紀錄。

## E. 特定標的附加要求

- **App**：取得行動應用基本資安檢測相關資安標章、符合無障礙設計。
- **網站**：導入 HTTPS。
- **涉個資**：資料落地（見 F）、個資保護措施。

## F. 資料落地與在地化

> 本案資料應儲存於中華民國境內；系統須支援繁體中文，包含台灣口音語音辨識、中英夾雜語句處理、符合台灣用語習慣之文字表達。

- **最常被直接翻譯國外範本忽略**，卻是能否在台灣上線的關鍵。

## G. 智慧財產權歸屬（提供可勾選選項，勿留空）

> 本案履約成果著作財產權歸屬：☐ 由機關取得　☐ 由廠商享有、機關取得授權利用。
> 著作人格權：☐ 廠商保留姓名表示權　☐ 其他約定＿＿。

- 原則：無特殊目的（國家機密 / 資安 / 高度客製）時，宜由廠商享有、機關取得授權（促進流通、降低成本）。需完整取得時於契約明確約定。
- **素材權利**：成果含音樂 / 圖片 / 影片片段時，須另取得公開傳輸 / 公開播送 / 公開上映授權；要求廠商驗收前提供授權書、著作財產權歸屬約定書（含其與員工之約定證明）。
- **分包 / 轉包著作權鏈條（最常落空、最難事後補救）**：得標廠商若聘第三方（分包商）協助完成履約標的，須於契約約定其**負有義務**替機關向該第三方取得授權或受讓著作財產權，並於決標 / 簽約日起一定期間（建議 10 日）內，將授權書、著作財產權歸屬約定書、讓與書等證明文件交付機關收存。**員工約定無法涵蓋分包商，須另立條款**——曾有機關取得得標廠商著作權，卻因部分程式轉包、第三方未讓與，致無法取得完整著作財產權。

## H. 驗收、保固與服務水準

> 廠商應於完成履約標的並通過驗收後請領款項；驗收付款前繳納保固保證金。保固期間【__】個月，涵蓋錯誤修正、功能調整與技術支援。
> SLA：系統可用率 ≥【99.9%】、API 回應時間 ≤【200ms】、未達標罰則【__】。

- 履約標的內容須詳載於契約及需求規格說明書；每條需求對應可驗收標準。

## I. 變更管理與變更請求流程（在 RFP 階段就先講清楚）

> 履約期間之需求變更應依正式變更請求（Change Request）流程辦理：提出者填具變更單，載明變更內容與原因、對範疇 / 時程 / 成本 / 風險之影響評估、不變更之後果與替代方案，經雙方變更控制窗口書面核准後始得執行；未經核准之變更不納入履約範圍。

- 變更請求分兩類：**範圍內變更**（客戶主導、原契約涵蓋）與**範圍外變更**（須另議時程 / 報價）。RFP 階段就揭露此流程，是抵禦範疇蔓延的核心機制。
- **為何放進 RFP**：委外失敗的首要根因是需求模糊與不受控的範疇蔓延；事前未約定變更流程，後期任何「再加一點」都會演變成延誤、超支與爭議。

## J. 退場、資料返還與移交（contract exit / transition-out）

> 契約屆滿、終止或解除時，廠商應於約定期限內：(1) 以雙方議定之開放格式完整返還機關所有資料與衍生資料，**並於返還確認後刪除 / 銷毀廠商端所有留存副本（含備份）並出具刪除 / 銷毀證明**，且不得保留或主張資料留置權；(2) 提供系統原始碼 / 設定 / 部署文件 / 維運手冊（依智財歸屬約定）；(3) 配合辦理知識移轉與服務移交，協助後手廠商或機關自行接管，確保服務不中斷。退場 / 移交之範圍、期限與配合義務應於契約明定。

- **為何放進 RFP**：缺退場與資料返還條款，機關易被廠商以資料 / 原始碼「綁架」，造成更換廠商困難、議價弱勢；CLM（契約生命週期管理）最佳實務要求把退場與資料返還條款前置設計。公部門與關鍵系統尤須具備。

## K. 專案治理與供應商盡職調查（把失敗預防框架寫成可投標條款）

> 機關側應指定產品負責人與技術窗口，並設能於【24】小時內裁決之專案發起人；履約期間採固定溝通節奏（每日 / 每週進度同步、共享專案管理工作區）。投標廠商應於建議書中：(1) 提出將**實際參與本案**之團隊名單、角色與資歷（非僅銷售或掛名人力）；(2) 視標的規模配合概念驗證（PoC）或啟動期付費技術評估，再進入完整開發。

- **為何放進 RFP**：常見失敗是「以精緻提案與低價得標，實際指派資格不足團隊」。要求揭露實際團隊、安排 PoC / 技術評估、明定甲方決策窗口與溝通儀式，能把治理從事後補救轉為投標即可檢核的條款。

## L. 計費方式與費用結構（資訊服務委外｜《機關委託資訊服務廠商評選及計費辦法》）

「計價及付款方式」是 §5 的法定應載事項。本辦法提供三種法定計費法及硬上限，與國際 Fixed-Price / T&M / Cost-plus 一一對應。**請依個案於招標文件與契約擇一訂明。**

> 本案服務費用採下列方式之一（於契約訂明）：
> ☐ 總包價法 / 單價計算法　☐ 按月 / 按日 / 按時計酬法　☐ 服務成本加公費法

- **總包價法 / 單價計算法**：適用工作範圍及內容**明確**、總價可正確估計或可按單價計算者。（對應國際 Fixed-Price；範圍鎖定時用。）
- **按月 / 按日 / 按時計酬法**：適用工作範圍小，或範圍 / 內容**無法明確界定**、總費用難以正確估計者。（對應國際 Time & Materials；範圍模糊時用。）
- **服務成本加公費法（§16）**：適用計畫性質複雜、費用不易確實預估或**履約成果不確定**者。費用＝直接費用（直接薪資＋管理費用＋其他直接費用）＋公費＋營業稅，並受下列**法定上限**：
  - 管理費用**不得超過**直接薪資（扣除非經常性給與之獎金後）之 **100%**。
  - 公費應為**定額**，且**不得超過**（直接薪資扣除非經常性給與之獎金＋管理費用）之 **25%**。
- **獎勵性報酬（§17，採成本加公費法時得約定）**：費用降低者，給付不逾所減省契約價金之 **50%**；績效提高者，給付不逾契約價金總額（或上限）之 **10%**。
- **預付款**：以不逾契約價金總額之 **30%** 為原則。
- **物價調整**：履約期間一年以上者，得自第二年起隨物價指數調整契約價金，並於招標文件載明調整項目、方式及上限。

> **選擇準則**：範圍明確 → 總包價；範圍模糊 → 計時；成果不確定 → 成本加公費。大型探索性案件可採混合（探索期固定價、建置期計時、按里程碑付款）。所有數值（100% / 25% / 50% / 10% / 30%）以全國法規資料庫最新版為準。

## M. 個人資料處理與事故通報（個資法遵）

僅寫「依個資法辦理」不可執行。應於 RFP / 契約具體載明：

> 廠商處理本案個人資料應遵循下列要求：
> - **資料類型與處理目的**：明列蒐集之個資項目、處理 / 利用目的，且不得逾越目的必要範圍（最小必要原則）。
> - **存取控制與留存**：以最小權限原則控管存取；明定資料留存期間，逾期或目的消失後依約刪除 / 銷毀。
> - **委託處理義務**：廠商為受託機構，應依個資法及主管機關規定為適當安全維護；再委託（分包）須經機關書面同意並課以同等義務。
> - **事故通報時限**：發生個資外洩 / 遭竊 / 竄改等事故時，應於約定時限（建議 72 小時內，並依機關內規）通報機關，協助查明、通知當事人及採行補救措施。

- **法規時效提醒**：個資法近年有修正，部分條文施行日期與子法尚有變動；引用具體條號 / 罰則時應標註版本與查核日，並提示由機關法務確認。

## N. 跨境資料與雲端區域揭露

僅寫「資料儲存於境內」不足以涵蓋備份 / 災備 / 傳輸路徑。建議要求廠商揭露資料地理流向：

> 廠商應於建議書揭露並於契約承諾：
> - **資料區域**：主要資料區、備份區、災難備援區之地理位置（是否均位於我國境內）。
> - **傳輸路徑與跨境**：資料傳輸是否經由境外節點；如有跨境，列明國家 / 地區、適用法域與法律風險。
> - **雲端服務**：所用公有雲服務之區域（region）、是否使用符合政府規範之雲端服務。
> - **加密與返還**：傳輸中與儲存中之加密方式；契約終止時之資料返還與刪除 / 銷毀證明（連動 J、M 節）。

- **適用情境**：涉及大量個資、機敏或關鍵基礎設施之系統，資料落地與跨境揭露宜列為 Blocker 級要求。

## O. AI 系統條款（生成式 / 預測式 AI、模型類服務）

採購含 AI / 機器學習之系統時，僅寫「導入 AI」不可驗收。應於 RFP / 契約具體載明：

> 廠商提供之 AI 功能應於建議書揭露並於契約承諾：
> - **模型與版本**：所用模型名稱、版本、供應商與部署位置（地端 / 私有雲 / 公有雲 API）；**模型版本變更或停用前之通知義務與期限**。
> - **評估指標與測試集**：交付前以雙方議定之測試資料集與評估量表（如準確率 / 召回 / F1 / 任務完成率）驗證；驗收標準載明可量測門檻，不得以「視情況」帶過。
> - **錯誤與幻覺控制**：揭露錯誤率 / 幻覺率量測方式與可接受上限；高風險輸出（法律 / 財務 / 個資）應有**人工覆核（human-in-the-loop）**流程與可追溯紀錄。
> - **輸入資料治理**：機關輸入之資料、prompt、對話紀錄**是否用於模型訓練 / 微調**（原則不得，例外須書面同意）；prompt / log / 對話之保存期限、存放位置與刪除機制。
> - **公平性與可解釋性**：對影響民眾權益之決策，要求偏誤（bias）評估、公平性檢核與可解釋性說明；保留稽核軌跡。

- **為何放進 RFP**：AI 採購的不確定性高，宜採**結構化 PoC / 付費技術評估**（連動 K 節）先驗證可行性與評估指標，再進入完整開發；評選須採**一致評估量表**避免各廠商自說自話。不得在 RFP 臆造模型效能數值，以「待驗證指標」標記。

## P. 資料治理條款（資料平台 / 資料倉儲 / 跨系統整合）

涉及大量資料蒐集、清理、整合或開放之系統，應要求可治理、可追溯：

> 廠商應提供並於契約維護：
> - **資料字典（data dictionary）與資料血緣（lineage）**：欄位定義、來源、轉換邏輯與流向，確保可追溯與可交接。
> - **資料品質 SLA**：完整性 / 正確性 / 即時性之量化門檻與量測方式；不達標之處理。
> - **權限矩陣（access matrix）**：以最小權限原則定義角色與資料存取範圍，定期審查。
> - **去識別化 / 假名化**：對外提供或分析用途之個資，依目的採去識別化 / 假名化，並評估再識別風險（連動 M 節）。

- **為何放進 RFP**：資料治理缺漏會在上線後才爆發（資料對不上、無法稽核、交接困難）；事前要求字典與血緣，是退場可交接（J）的前提。

## Q. 雲端韌性與營運持續（RTO / RPO / DR / 金鑰）

N 節揭露資料**在哪**；本節要求服務**斷得掉、回得來、鑰匙誰管**：

> 廠商應於建議書承諾並於契約量化：
> - **RTO / RPO**：各關鍵服務之復原時間目標（RTO）與資料復原點目標（RPO）數值。
> - **BCP / DR 演練**：營運持續與災難復原計畫，**並定期實地演練**（如每年至少一次）並提供演練報告。
> - **金鑰管理**：加密金鑰之管理方式（KMS / BYOK 自帶金鑰 / HYOK），機關對金鑰之掌控程度；委外不得使機關喪失對自身資料之解密控制權。
> - **備份不可竄改性（immutability）**：關鍵備份具防勒索 / 防竄改特性與保留期；備份還原需定期驗證。

- **適用情境**：關鍵業務、對外服務、涉個資或金流之系統，RTO / RPO 與 DR 演練宜列為必要要求；缺之屬 Major 以上。

## R. 開源與供應鏈安全（SBOM / 授權 / 弱點）

軟體供應鏈攻擊與授權合規是現代資訊採購的標準要求：

> 廠商交付之軟體應隨附並於契約維護：
> - **SBOM（軟體物料清單）**：交付物所含開源 / 第三方元件清單與版本，供機關掌握成分與風險。
> - **授權合規（license compliance）**：所用開源元件之授權型態揭露，確保不牴觸機關使用 / 散布需求（與 G 智財歸屬連動，避免 copyleft 污染機關自有程式）。
> - **相依弱點修補 SLA**：對已知漏洞（含開源相依）之修補時限（如 critical 於 N 日內）；提供弱點掃描 / SCA 報告。
> - **容器 / 映像掃描**：以容器交付者，映像須通過弱點掃描並移除高風險項目後始得部署。

- **為何放進 RFP**：得標後才發現元件授權不相容或帶已知漏洞，補救成本極高；SBOM 與修補 SLA 把供應鏈風險前移到投標即可檢核。
