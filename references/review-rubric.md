# RFP 審查 rubric（review 模式的 WHAT to check）

review 模式逐項套用本 rubric，輸出**依嚴重度分級的 findings**。HOW（流程與輸出格式）在 SKILL.md。

## 審查心法（寫 finding「為什麼重要」時的依據）

- **RFP 是風險定價的資訊揭露機制**：每個沒寫清楚的條款，誠實廠商會加風險溢價、投機廠商會低價搶標再靠變更賺回；模糊不會更便宜，只把成本推遲到爭議。→ 指出缺漏時，講清楚它如何推高報價或埋下爭議，比泛泛說「不完整」有力。
- **可驗收性貫穿全程**：寫不出驗收標準的需求＝還沒想清楚的需求。缺驗收標準幾乎都是 Blocker。
- **最有利標 ≈ 成果導向採購前身**：評選規則一旦公告不得變更（剛性）；遇到要求敏捷 / SOO 的政府案，提醒剛性與彈性的張力。

## 嚴重度定義

- **Blocker（阻擋）**：會導致採購無效、法律風險或必然爭議。例：評選委員會組成不合法、評選配分未載明、缺驗收標準、缺智財歸屬約定。
- **Major（重大）**：顯著降低提案品質或可比較性、易引發超支 / 變更。例：需求模糊、範疇未界定含 / 不含、預算完全不揭露、無釋疑機制。
- **Minor（輕微）**：可改善但不致命。例：用語不一致、缺里程碑細分、無 PoC 安排。

## 審查清單（逐項判定：符合 / 缺漏 / 不足）

### 1. 文件定位與範疇
- [ ] 用對文件（RFP 而非 RFQ；得標後事項未誤置於 RFP）。
- [ ] 工作範疇明確劃分「包含 / 不包含」（防範疇蔓延）。— 缺 = Major
- [ ] 業務背景、痛點、量化目標清楚。

### 2. 需求品質
- [ ] 功能需求分級（必要 / 應有 / 加分）。
- [ ] 每條需求可驗證、對應驗收標準。— 缺驗收標準 = Blocker
- [ ] 非功能需求量化（效能 / 安全 / 可用 / 可維護）。— 缺 = Major
- [ ] （維護 / 擴充案）系統測試上線、新舊系統轉移、技術移轉需求明列。— 缺（且屬維護 / 擴充案）= Major（與「退場 / 資料返還」不同，此為履約中轉換）

### 3. 台灣法制與程序（政府採購案）
- [ ] 招標 / 決標方式選定且適切（異質採購→最有利標）。
- [ ] 評選項目、配分、權重於文件明確載明。— 缺 = Blocker
- [ ] **價格配分 / 權重落在 20%–50%**（《最有利標評選辦法》§16 評分、§17 序位評比）。— 非固定價格案低於 20% 或高於 50% = Blocker（明文禁止，決標可被爭議）。**例外**：招標文件訂明**固定價格給付**者，價格得低於 20%，不在此限——判定前先確認是否為固定價格案，避免誤報。
- [ ] 簡報及現場詢答若列為評選項目，配分 / 權重 ≤ 20%，且載明簡報不得更改投標文件（§10/§14 範圍）。— 逾 20% = Major
- [ ] 評選委員會組成合法（5 人以上、外聘 ≥ 1/3、召集人一級主管以上）。— 違反 = Blocker
- [ ] 等標期符合金額級距；釋疑機制明定。— 無釋疑 = Major
- [ ] （資訊服務案）對照《機關委託資訊服務廠商評選及計費辦法》§5 法定應載事項與 §7 評審項目（見 `taiwan-procurement.md` §8）。— 缺關鍵款項 = Major

### 4. 資安與在地化（見 clause-library.md）
- [ ] 資安分級與對應控制。— 公部門缺 = Blocker
- [ ] 禁陸製 / 陸籍、SSDLC、資安標章 / 無障礙 / HTTPS（依標的）。
- [ ] 資料落地、繁中在地化。— 缺 = Major

### 5. 契約、驗收、智財、保固
- [ ] 履約 / 驗收 / 付款條件前置揭露。
- [ ] 智財歸屬提供可勾選選項（未留空）、含素材授權。— 缺 = Blocker
- [ ] 若可能分包 / 轉包，已約定分包商著作權之取得義務與證明文件交付期限。— 缺（且確有分包）= Major
- [ ] （資訊服務案）計價及付款方式擇定法定計費法（總包價 / 計時 / 成本加公費），成本加公費案載明公費 ≤25%、管理費 ≤100% 等上限。— 缺 = Minor
- [ ] 保固期間與維護範圍、SLA 與罰則。

### 6. 變更管理與退場（最常被整章漏掉）
- [ ] 變更管理 / 變更請求流程於 RFP 揭露（提出 / 影響評估 / 書面核准）。— 缺 = Major（範疇蔓延的主因）
- [ ] 退場、資料返還與移交條款（資料以開放格式返還、不主張留置、配合後手接管）。— 公部門 / 關鍵系統缺 = Major

### 7. 治理與盡職調查
- [ ] 機關側決策窗口 / 溝通節奏明定。
- [ ] 要求廠商揭露實際參與團隊名單與資歷（非掛名人力）。— 缺 = Minor
- [ ] 視標的安排 PoC / 啟動期技術評估。— 大型 / 高風險案缺 = Minor

### 8. 評選透明與流程
- [ ] 評選標準於 RFP 內揭露。
- [ ] 預算範圍揭露（利於技術 / 價值取捨）。— 完全不揭露 = Major
- [ ] 提交格式、頁數、時程、聯絡窗口完整。

## RFP 八大常見錯誤（review 對照表）

1. 範疇與要件不明確 → 廠商以假設填空。
2. 過度聚焦功能清單，忽略要解決的業務問題。
3. 忽略交付與協作現實（時程、產能、決策者、工作方式）。
4. 不分享預算範圍。
5. 易選錯夥伴（只看銷售簡報，未盡職調查）。
6. 溝通落差（時區、文化、非同步）。
7. 輕忽法律 / 契約（聯合協議、智財、爭議解決）。
8. 必要要件回應機制不足（必要要件＝合格 / 不合格，非評分）。

## 委外失敗根因（findings 的「為什麼重要」可引用）

- 首要：**需求模糊＋溝通不良**；一份雙方解讀不同的高層簡報 → 做出錯的東西。
- 次要：**範疇蔓延**（客戶初期不確定、不了解變更後果）→ 延誤 / 超支 / 品質下降。
- 預防：需求鎖定、正式治理（角色 + 24h 決策者）、供應商盡職調查、結構化溝通、**正式變更請求流程**，並在 RFP 階段就寫入**變更管理與退場 / 資料返還**條款。

## Finding 輸出格式（每項）

```
[嚴重度] 章節/位置 — 問題簡述
- 為什麼重要：<風險 / 後果，可引用根因或法制>
- 建議修正：<具體、可貼入的改寫或補充；指向 clause-library.md 條款>
```
